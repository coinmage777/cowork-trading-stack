### Watchdog

systemd + 별도 watchdog 스크립트를 같이 사용합니다. systemd가 프로세스 죽음만 감지한다면, watchdog은 "프로세스는 살아있지만 hung" 상태를 감지합니다.

```bash
# watchdog.sh
#!/bin/bash
LOG=/home/bot/perp-bot/logs/bot.log
LAST_MOD=$(stat -c %Y "$LOG")
NOW=$(date +%s)
DIFF=$((NOW - LAST_MOD))

if [ $DIFF -gt 300 ]; then
    # 5분 동안 로그 없음 = 좀비
    pkill -f multi_runner.py
    sleep 5
    systemctl restart perp-bot
    echo "$(date) restarted (stale)" >> /home/bot/perp-bot/logs/watchdog.log
fi
```

cron으로 1분마다:
```
* * * * * /home/bot/perp-bot/watchdog.sh
```

## 로깅 / 모니터링

### 구조화 로그

평문 로그는 grep만 가능합니다. 구조화 로그(JSON)는 분석이 쉽습니다.

```python
import logging
import json

class JsonFormatter(logging.Formatter):
    def format(self, record):
        log_obj = {
            "ts": self.formatTime(record),
            "level": record.levelname,
            "logger": record.name,
            "msg": record.getMessage(),
        }
        if hasattr(record, "exchange"):
            log_obj["exchange"] = record.exchange
        if hasattr(record, "trade_id"):
            log_obj["trade_id"] = record.trade_id
        return json.dumps(log_obj, ensure_ascii=False)

handler = logging.FileHandler("logs/bot.json")
handler.setFormatter(JsonFormatter())
logger = logging.getLogger("bot")
logger.addHandler(handler)

logger.info("position opened", extra={"exchange": "hyperliquid", "trade_id": 12345})
```

### Equity Tracker

핵심 인프라입니다:

```python
async def equity_tracker_loop():
    while True:
        for exchange in exchanges:
            try:
                balance = await exchange.get_collateral()
                db.insert_equity_snapshot(
                    timestamp=datetime.utcnow().isoformat(),
                    exchange=exchange.name,
                    balance_usd=float(balance),
                )
            except Exception as e:
                logger.error(f"equity tracker {exchange.name}: {e}")
        await asyncio.sleep(600)  # 10분
```

이것으로 일일 / 주간 / 월간 PnL을 계산합니다. **로그 PnL과 비교 절대 X. 잔고 스냅샷이 진실입니다**.

## 재시작 / 핫 리로드

### Graceful Restart

봇이 포지션을 열고 있는 상태에서 재시작하면 상태를 잃을 수 있습니다. 패턴:

1. **State Manager**: DCA / trailing 상태를 매 변경 시 JSON에 저장
```python
def save_state(state: dict):
    with open("trader_state.json", "w") as f:
        json.dump(state, f)
```

2. **재시작 시 복원**:
```python
def load_state():
    if Path("trader_state.json").exists():
        with open("trader_state.json") as f:
            return json.load(f)
    return {}
```

3. **Orphan 방지**: 재시작 시 거래소 실제 포지션 조회 → DB의 open trade와 일치하지 않으면 close 처리

### Hot Reload

config 변경할 때 봇 재시작 없이 파라미터만 갱신합니다:

```python
import signal

def reload_config(signum, frame):
    logger.info("Reloading config")
    config.load()  # 다시 읽음
    apply_to_strategies()

signal.signal(signal.SIGHUP, reload_config)
```

```bash
kill -HUP $(pgrep -f multi_runner.py)
```

Windows는 SIGHUP이 안 됩니다 → 파일 트리거 패턴 (4장 참조).

## 운영 원칙 — 시행착오로 배운 것

### 1) 변경 → 검증 → 스케일

새 전략 / 파라미터는 다음 순서로 진행합니다:
1. 백테스트 통과
2. 페이퍼 트레이드 1주
3. 소액 라이브 (자본의 5%)
4. 1주 모니터링 → 문제 없으면 사이즈 확대
5. 절대 한 번에 풀 사이즈로 박지 않습니다

### 2) 한 번에 하나만 변경

5개 파라미터를 동시에 변경하면 어느 것이 효과가 있는지 알 수 없습니다. A/B 테스트 식으로 진행합니다.

### 3) 데이터 기반 결정

"감으로 손절 줄여볼까" 금지. DB에서 실제 데이터를 보고 결정합니다. 매번 변경 전에 손절 분포 / 승률 분포 / DCA 깊이별 승률 등을 확인합니다.

### 4) 서킷브레이커 필수

일일 -X% 손실 시 자동 정지합니다. 무한 손실을 막는 안전망입니다. 기본값은 자본의 5–10%입니다.

### 5) 알림 우선순위

- CRITICAL (즉시): 일일 X% 손실, 거래소 잔고 0, 봇 크래시 → 텔레그램 + 모바일 푸시
- WARNING (즉시): 거래소 비활성화, WS 끊김 → 텔레그램만
- INFO (1일 1회): 일일 PnL 요약 → 텔레그램만
- DEBUG: 모든 진입 / 청산 → 로그만

알림이 너무 많으면 무뎌집니다. 정말 중요한 것만 받습니다.

### 6) Post-Mortem 작성

큰 손실 / 큰 사고 발생 시 reproducible 형식으로 기록합니다:

```markdown
# Post-Mortem: 27일 연속 손실

## What happened
27일 연속 일일 -1~3% 손실 발생.

## Why
TP 0.4% / SL 3%, 본전 승률 94% 필요. 실제 승률 70%대 → 구조적 손실.
원인: pnl_percent가 마진 기준으로 잘못 디자인됨. R:R 수학 검증 안 함.

## How fixed
- TP 0.4% → 2%, SL 3% → 2.5%, leverage 15x → 10x
- 본전 승률 55%로 낮춤
- 실제 승률 67~84% 확인 → 수익성 전환

## How to prevent
- 모든 새 전략은 본전 승률 < 60%인지 검증
- TP / SL은 마진 기준 X, R:R 기준으로 디자인
- 백테스트 PF > 1.5 필수
```

이런 문서가 쌓이면 자산입니다. 같은 실수를 두 번 하지 않습니다.
