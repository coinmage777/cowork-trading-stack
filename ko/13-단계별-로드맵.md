# 13. 단계별 로드맵

0에서 지금까지 온 길을 시간 / 단계로 정리한 내용입니다. 경험 기반이며 정답은 아닙니다. 각자 속도에 맞춰 진행하시기 바랍니다.

## 0. 시작 전

자본: 작게. 처음 거래소당 $50~100으로 시작합니다.
시간: 주말 + 평일 저녁 1~2시간.
지식: Python 기초 + 차트 보기 정도.

## 1주차 — 환경 셋업

### 목표
도구가 손에 익는 것.

### 할 것
- [ ] VSCode / Cursor 설치
- [ ] Python 3.10+ + venv
- [ ] Git + GitHub
- [ ] Claude API 키 발급
- [ ] Telegram 봇 만들기
- [ ] 거래소 1개 (Bybit 또는 Binance 추천 — 안정적) 계정 생성
- [ ] API 키 발급 (트레이딩만, 출금 X)
- [ ] `.env` 파일에 키 저장 → `.gitignore` 추가
- [ ] ccxt로 잔고 조회 스크립트

### 첫 코드
```python
# balance_check.py
import os
import asyncio
from dotenv import load_dotenv
import ccxt.async_support as ccxt

load_dotenv()

async def main():
    ex = ccxt.bybit({
        "apiKey": os.getenv("BYBIT_API_KEY"),
        "secret": os.getenv("BYBIT_SECRET"),
        "options": {"defaultType": "swap"},
    })
    balance = await ex.fetch_balance()
    print(f"USDT: {balance['USDT']['total']}")
    await ex.close()

asyncio.run(main())
```

이것이 작동하면 1주차 합격입니다.

## 2주차 — 첫 자동화

### 목표
가격 모니터링 + 알림 봇.

### 할 것
- [ ] 가격을 1분마다 폴링하는 봇
- [ ] 특정 가격 도달 시 텔레그램 알림
- [ ] 백그라운드 실행 (tmux / screen)

### 코드
```python
import asyncio
import aiohttp
import os

async def notify(text):
    token = os.getenv("TELEGRAM_BOT_TOKEN")
    chat_id = os.getenv("TELEGRAM_CHAT_ID")
    url = f"https://api.telegram.org/bot{token}/sendMessage"
    async with aiohttp.ClientSession() as s:
        await s.post(url, json={"chat_id": chat_id, "text": text})

async def price_alert(target_price: float):
    ex = ccxt.binance()
    sent = False
    while True:
        ticker = await ex.fetch_ticker("BTC/USDT")
        if not sent and ticker["last"] >= target_price:
            await notify(f"BTC hit ${target_price}")
            sent = True
        await asyncio.sleep(60)

asyncio.run(price_alert(70000))
```

## 3~4주차 — 볼륨 파밍 봇

### 목표
첫 트레이딩 봇. 양방향 헷지로 볼륨만 누적.

### 할 것
- [ ] 5장 (Volume Farmer) 의 단순 버전 구현
- [ ] 한 거래소만, 한 자산만 (BTC)
- [ ] 매우 작은 사이즈 ($10~20)
- [ ] 1주일 운영 → PnL 분석

### 검증 기준
- 양방향 PnL 합이 -1% 이내 (수수료 + 슬리피지 감안)
- 봇이 24시간 끊김 없이 돌아감
- 텔레그램 알림 정상 작동

## 5~8주차 — 옵시디언 + 메모리 시스템

### 목표
지식 인프라 구축. 트레이드 / 리서치 / 콘텐츠 누적 시작.

### 할 것
- [ ] 옵시디언 설치
- [ ] 볼트 폴더 구조 설정 (4장 참조)
- [ ] 매매 일지 템플릿 작성
- [ ] 매일 매매 / 봇 운영 기록
- [ ] CLAUDE.md 작성 (컨텍스트)
- [ ] MemKraft 셋업 (옵션)
- [ ] 첫 프로젝트 리서치 노트 작성 (관심 있는 거래소 / 토큰)

이 시점에서 단순 매매가 아니라 시스템적 사고가 시작됩니다.

## 9~12주차 — 페어 트레이딩 1단계

### 목표
백테스트 + 페이퍼 트레이드.

### 할 것
- [ ] BTC vs ETH 1년 데이터 수집 (1분봉)
- [ ] 단순 z-score 시그널 백테스트 (8장 참조)
- [ ] 메트릭 계산 — Profit Factor, Win Rate, MDD
- [ ] 파라미터 sensitivity 체크
- [ ] 페이퍼 트레이드 (실시간 가격, 가짜 잔고)
- [ ] 페이퍼 vs 백테스트 결과 비교

### 검증 기준
- 백테스트 PF > 1.5
- 페이퍼 트레이드 결과가 백테스트와 ±20% 이내

## 13~16주차 — 페어 트레이딩 라이브

### 목표
소액 라이브 + 안정 운영 검증.

### 할 것
- [ ] 한 거래소 ([Hyperliquid](https://miracletrade.com/?ref=coinmage) 추천 — 메인) 라이브
- [ ] 사이즈 매우 작게 (자본의 5%)
- [ ] 모든 트레이드 DB 로깅
- [ ] 일일 PnL 리포트 (cron / daily_report.py)
- [ ] 서킷브레이커 (-X% 일일 손실 시 정지)
- [ ] 1주일 운영 → 백테스트와 비교

### 발생할 수 있는 문제
- 슬리피지가 백테스트 가정보다 큼
- 봇 크래시 → 좀비 포지션
- 거래소 API 변경 / rate limit
- 시장 환경 변화 (variance regime change)

이 시점에서 11장 (운영 인프라) 본격 적용이 시작됩니다.

## 17~24주차 — 멀티 거래소 / 고도화

### 목표
거래소 N개 동시 + 시그널 다양화.

### 할 것
- [ ] 거래소 2~3개 추가 (12장 참조)
- [ ] 거래소 그룹 분산 (시차 진입)
- [ ] 추가 시그널 — 모멘텀 / 볼린저 / RSI
- [ ] 레짐 필터 (correlation 기반)
- [ ] 트레일링 스탑
- [ ] DCA (조심스럽게)

### 검증 단계
각 변경 후:
1. 백테스트
2. 페이퍼 1주
3. 라이브 소액 1주
4. 안정 확인 시 풀 사이즈

한 번에 하나씩 진행합니다.

## 1년차 끝

이 시점에서 운영자는 다음 상태에 도달합니다.
- 봇 1~2개를 안정 운영
- 매일 PnL 자동 리포트
- 새 거래소 / 시그널 추가에 1~2일이면 충분
- 운영 사고 / 손실 패턴 자체 분석 가능
- 알파 가설 + 검증 시스템 보유

수익 규모는 케이스 바이 케이스입니다. 처음 1년은 거의 본전인 경우가 많습니다. 본격적인 수익은 2~3년차부터 나타납니다.

## 2~3년차 — 스케일

### 목표
자본 늘리기 + 새 영역.

### 후보
- 자본 키우기 (검증된 전략에만)
- 새 시장 (예측시장 / 옵션 / DeFi yield)

추천하는 것은 한 가지입니다. **수익이 늘어도 자본을 다 박지 마세요.** 자본은 천천히 늘리고, 다양화로 리스크를 분산합니다.

## 흔한 실수 — 했던 사례

### 1) 한 번에 풀 사이즈
검증되지 않은 전략에 풀 사이즈 → 큰 손실. 항상 5% → 25% → 50% → 100% 단계적으로 진행합니다.

### 2) 백테스트 안 함
"감으로 좋아 보여서" 라이브 → 27일간 손실이 발생한 사례가 있습니다. 백테스트를 한 시간만 했어도 알 수 있는 문제입니다.

### 3) 알파 누설
"내 봇이 이렇게 한다"고 자랑하면 알파가 죽습니다. 큰 디테일은 적지 않습니다 (이 가이드도 framework만 다룹니다).

### 4) 너무 많은 거래소 / 전략
한 번에 3개의 새 거래소를 추가하면 디버깅 지옥에 빠집니다. 한 번에 하나씩 진행합니다.

### 5) 모니터링 안 함
"봇 잘 돌아가겠지" → 며칠 후 손실 발견. 매일 5분이라도 확인합니다.

### 6) 데이터 안 봄
DB를 들여다보지 않고 감으로 결정 → 항상 틀린 곳에 시간을 쓰게 됩니다.

### 7) 좀비 프로세스
봇 재시작 후 이전 인스턴스가 살아있어서 중복 주문 → 큰 손실. 항상 ps aux로 확인합니다.
